# Skill: Unit Tests — FIRST Principles

## Purpose

This skill defines the **FIRST** properties that every unit test generated by
the Unit Test Generator agent MUST satisfy.  The agent MUST read this skill
before writing any test code and MUST include a FIRST compliance assessment
in `test-report.md`.

---

## The FIRST Properties

### F — Fast

> A unit test must complete in milliseconds, not seconds.

**Criteria:**
- No network calls (no HTTP requests, no socket connections).
- No disk I/O beyond what the unit under test strictly requires.
- No `time.sleep()` or polling loops in the test body.
- No database migrations or schema creation (use in-memory or mocked stores).

**How to verify:** Run the suite with `pytest --tb=short -q`.  The total wall-clock
time for the generated tests must be < 100 ms per test on a commodity laptop.

---

### I — Independent

> Tests must not depend on each other or on shared mutable state.

**Criteria:**
- Each test sets up its own preconditions using fixtures or setup methods.
- No test reads state written by a previous test.
- Tests can run in any order and produce the same result.
- Use `autouse` fixtures (e.g. `reset_expenses`) to ensure a clean slate.
- Never rely on insertion order of a shared collection.

**How to verify:** Run the suite twice — once in default order, once with
`pytest --randomly-seed=12345` (if `pytest-randomly` is installed).  Results
must be identical.

---

### R — Repeatable

> A test must produce the same result on every run, in every environment.

**Criteria:**
- No reliance on system time, random values, or environment variables unless
  they are explicitly seeded or mocked.
- No dependency on network availability.
- No assumption about the order of items in sets or dicts (use sorted comparisons).
- Hardcode expected values rather than re-computing them in the assertion.

**How to verify:** Run the suite 3 times in a row on a clean checkout.  All
three runs must report the same pass/fail result.

---

### S — Self-validating

> A test must report a binary outcome (PASS / FAIL) without human inspection.

**Criteria:**
- Every test MUST contain at least one `assert` statement.
- Assertions use specific matchers: `pytest.approx` for floats,
  `assert x == expected` for exact values, `pytest.raises` for exceptions.
- No `print()` used as a substitute for an assertion.
- The failure message (via `pytest.fail` or assertion rewriting) must describe
  *what* was wrong without requiring the reader to look up source code.

**How to verify:** Introduce a deliberate off-by-one in the source and confirm
that at least one test fails with a clear message.

---

### T — Timely

> Tests are written at the same time as (or immediately after) the code they cover.

**Criteria:**
- Generated tests target the functions changed in the current pipeline run (as
  listed in `fix-summary.md`) — not legacy code unchanged by this run.
- Each generated test references the specific bug ID or fix it exercises
  (in the test name or docstring): e.g. `test_total_single_expense_bug001_fix`.
- Tests must not duplicate coverage already provided by `test_expense_tracker.py`.

**How to verify:** Cross-reference the list of changed functions in
`fix-summary.md` against the set of functions covered by the generated tests.
Every changed function must have at least one new test.

---

## Compliance Checklist (include in `test-report.md`)

```
## FIRST Compliance Assessment

- [ ] Fast        — all generated tests finish in < 100 ms each; no I/O / network
- [ ] Independent — autouse `reset_expenses` fixture ensures isolation; no ordering dependency
- [ ] Repeatable  — no random/time/env dependencies; deterministic assertions
- [ ] Self-validating — every test has ≥1 assert; failure messages are descriptive
- [ ] Timely      — tests target only functions changed in this pipeline run
```

For each property, replace `[ ]` with `[x]` if satisfied and add one sentence of evidence.

---

## Test Naming Convention

```
test_<function_name>_<condition>_<expected_outcome>
```

Examples:
- `test_get_total_single_expense_returns_full_amount`
- `test_add_expense_negative_amount_raises_value_error`
- `test_get_expenses_uppercase_category_matches_lowercase_stored`

---

## Forbidden Patterns

| Pattern | Reason |
|---------|--------|
| `import time; time.sleep(n)` | Violates Fast |
| `assert True` or empty test body | Violates Self-validating |
| Modifying a module-level list without cleanup | Violates Independent |
| `random.random()` without `random.seed()` | Violates Repeatable |
| Tests for functions not in `fix-summary.md` scope | Violates Timely |
